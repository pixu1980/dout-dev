# License Policy

This file is an operational proposal. Before the official open source launch, add real licenses to the repository.

## Why this is needed

Without clear licensing:

- code/content reuse is unclear
- contributors do not know what they grant
- forks, remixes, and distribution are ambiguous
- open source collaboration is weaker

## Recommended choice

For a community blog with code and articles, clearest option is dual licensing:

| Area | Recommended license |
| --- | --- |
| Source code | MIT or Apache-2.0 |
| Articles and editorial content | Creative Commons Attribution 4.0 International, CC BY 4.0 |
| Original visual assets | CC BY 4.0, unless exceptions are declared |
| Trademarks, logo, and brand | All rights reserved, unless explicit permission |

## Suggested README note

```md
## License

Code in this repository is licensed under [TODO: MIT or Apache-2.0].

Articles and editorial content are licensed under [TODO: CC BY 4.0], unless otherwise stated.

Project name, logo and brand assets are not covered by the open source license and may not be used to imply endorsement without permission.
```

## Suggested CONTRIBUTING note

```md
By contributing to this repository, you agree that your contributions are licensed under the licenses used by the project:
- code contributions under the code license
- content contributions under the content license
```

## Exceptions

An author can propose content under a different license only if:

- license is declared in frontmatter
- maintainers approve the exception
- it does not create conflict with publication, distribution, or site archiving

## Third-party assets

Third-party assets must include:

- author
- source
- license
- license link
- modifications made

If usage permission cannot be demonstrated, asset must not be included.
