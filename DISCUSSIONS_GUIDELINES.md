# GitHub Discussions Guidelines

Article comments for dout.dev are managed through GitHub Discussions.

This keeps the blog lightweight, transparent, and aligned with open source workflows.

## Goal of Discussions

Discussions are for:

- commenting on published articles
- asking author questions
- proposing corrections
- suggesting follow-ups
- sharing related experiences
- discussing technical trade-offs
- surfacing ideas for new articles or PRs

## Basic rules

When participating:

- stay on article topic
- critique ideas, not people
- explain your reasoning
- avoid aggressive sarcasm or provocation
- do not turn thread into framework/language/tool wars
- do not use Discussions for spam or self-promotion
- respect Code of Conduct

## Useful comments

A useful comment can be:

- a clear question
- a practical example
- a source-backed correction
- an alternative use case
- a different perspective
- a relevant link
- a reasoned doubt

## Non-useful comments

May be moderated:

- insults
- discriminatory comments
- persistent off-topic content
- flame bait
- spam
- repeated self-promotion
- copied AI comments without added value
- support requests unrelated to article
- escalation of personal conflicts

## Linking articles and Discussions

Each published article should have a linked Discussion.

Discussion should include:

- article title
- article link
- author
- short conversation prompt
- optional starter question

Template:

```md
# Discussion: [ARTICLE_TITLE]

Article:
[ARTICLE_LINK]

Author:
[AUTHOR]

What did you think?
- What part was most useful?
- Do you have a different take?
- Is there something worth expanding in a follow-up?
```

## Author role

Author is encouraged, not required, to answer every comment.

Expected:

- respond respectfully
- accept well-founded corrections
- avoid attacking people who disagree
- alert maintainers if moderation is needed

## Maintainer role

Maintainers can:

- move off-topic discussion
- close degraded threads
- hide or remove comments
- issue warnings
- stop repetitive conflict loops
- enforce Code of Conduct

## Escalation

If a Discussion becomes problematic:

1. public or private reminder
2. request to return on-topic
3. temporary thread closure
4. removal of problematic comments
5. formal report handling under Code of Conduct
6. account limitation if needed

## When to open issue instead of Discussion

Open issue when:

- you found site bug
- there is a broken link
- there is accessibility problem
- you want to propose site feature
- there is reproducible technical error

Use Discussion when:

- you want to discuss an article
- you want to ask open question
- you want to propose a theme
- you want to gather opinions
